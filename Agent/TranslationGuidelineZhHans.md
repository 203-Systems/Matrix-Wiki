# 简体中文（`zh-Hans`）翻译改动规范（Translation Guideline）

## 总原则

- 只修改“翻译问题”，不要做多余的改动（不重写段落、不顺手优化表达、不改无关格式/结构）。
- 尽量“一行一行”地改：用户指出哪一行/哪一句不对，就只改对应那一行/那一句。
- 保持原有文档结构与标记不变（例如 MDX 组件、链接、代码块、JSON 字段等），只替换文字内容。

## 具体翻译约定（来自本次对话）

- 应用名称统一标准：
  - `Performance` → `表演`（例如 `Performance 应用` → `表演应用`）
  - `Note` → `音符`（例如 `Note 应用` → `音符应用`）
  - `Companion`：不改名（保持 `Companion 应用`），不要删除该条目。
  - `Custom Control Map`：译为“自定义控制应用”（不使用“自定义控制映射应用”）。
- 文件夹命名与状态用语：
  - `应用文件大1 (活动)` → `应用文件夹1（当前）`
  - `应用文件大1 - 已创建并已选中。（长按编辑颜色或删除文件夹）` → `应用文件夹1 - 已创建并已选中。（长按编辑颜色或删除文件夹）`
  - 同类项保持一致：例如 `应用文件夹2（未选中）`、`应用文件夹3 ~ 6（未创建）` 等（仅在原文存在对应含义时替换）。
- 状态括号与用词：
  - 状态使用中文全角括号：`(未选择)` → `（未选中）`、`(已选择)` → `（已选中）`
  - `active` 语境优先译为“当前”（例如“应用文件夹1（当前）”、“当前应用”）。
- 中文与 MDX 标签之间不要额外空格：
  - `右侧为 <DocLink ...>` → `右侧为<DocLink ...>`（中间不加空格）。

## 维护要求（后续工作）

- 每次你（用户）提出新的翻译需求或做法要求时，都要把新增/变更的规则追加到本文件，保持可追溯。
- 不做无意义的“同义改写/文案润色”：不要在未被要求时改标题/告示等已可读中文（例如“应用启动器/应用程序启动器”来回替换）。
- 语法与口语化偏好（持续补充）：
  - 偏口语、直白：优先用“会”而不是“将”（例如“它会变亮”）。
  - 避免冗余词：例如尽量不用“任何”，可改为“点击空白区域取消选择”。
  - 术语更自然：优先用“应用”，尽量不用“应用程序”（除非原文明确区分）。
  - 不强行口语化交互提示：按钮/操作描述优先用“点击…/长按…”，避免“点一下…”这类过度口语化表述，除非用户明确要求。
  - 允许固定句式：例如 `- **选择应用**：点击应用以选择它（它会变亮，其他应用变暗）。点击空白区域取消选择。`
  - 避免重复信息：如果 UI 标题已说明（例如已显示“配置菜单”），描述不必重复强调标题，可写成“在这里调整各种选项”这类补充说明。
